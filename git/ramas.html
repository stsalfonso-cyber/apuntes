3. Ramas

Rama / Branch

Las ramas permiten que cada desarrollador (o cada tarea) viva en su propia realidad paralela.

1. Conceptos básicos

  • main (o master): Es la rama principal. Es la "realidad oficial". Todo lo que está aquí debe estar probado y funcionando.
  • Rama (Branch): Es un clon de main en un momento determinado. Al crear una rama, creas un espacio donde puedes hacer cambios sin afectar a la realidad oficial.

2 Los comandos más comunes


# 1. Crear una rama y saltar a ella inmediatamente
git checkout -b nombre-de-la-tarea

# 2. Listar ramas para saber dónde estás
git branch

# 3. Volver a la rama principal (antes de hacer un merge o pull)
git checkout main

(Nota: En versiones modernas de Git, git switch -c nombre-de-la-tarea es el comando preferido, pero checkout es el clásico que verás en todos sitios).

3 El Flujo de Trabajo (El "Workflow" de clase)

  1. ⚪ Fase 1: Sincronización y Creación

    1️⃣ Me pongo al día: git checkout main y luego git pull origin main.

    2️⃣ Ramifico: git checkout -b logo-spagnolo-nuevo.

  2. ⚪ Fase 2: Desarrollo

    3️⃣ Trabajo: Haces cambios en tus archivos.

    4️⃣ Guardo: git add . y luego

    git commit -m "feat: implementado nuevo logo Spagnolo".

  3. ⚪ Fase 3: Subida (Remoto)

    Ahora es cuando haces visible tu trabajo en GitHub.

    5️⃣ Subo mi rama: git push -u origin logo-spagnolo-nuevo. (Ahora tu rama ya vive en GitHub).

  4. ⚪ Fase 4: La "fusión" profesional (Pull Request)

    En un equipo profesional, no haces merge en tu ordenador. Sigues estos pasos en GitHub:

    6️⃣ Abres un Pull Request (PR): Entras en GitHub, verás un botón que dice "Compare & pull request". Lo pulsas, añades una descripción y pides que revisen tu trabajo.

    7️⃣ Revisión: Alguien (o tú mismo) revisa el código. Si todo está bien, pulsas "Merge pull request" en la web de GitHub.

    ✔ ¡Listo! Tus cambios ya están en main en el servidor.

  5. ⚪ Fase 5: Limpieza Local

    Ahora que main en GitHub tiene tus cambios, tu main local está desactualizado. Vamos a ponerlo al día y limpiar.

    8️⃣ Vuelvo a main: git checkout main

    9️⃣ Sincronizo el éxito: git pull origin main. (Esto descarga el merge que hiciste en el paso 7).

    🔟 Borro mi rama local: git branch -d logo-spagnolo-nuevo.


Resumen: "El flujo DAW profesional es: Pull(main) -> Branch -> Commit -> Push(branch) -> PR(GitHub) -> Merge(GitHub) -> Pull(main) -> DeleteBranch(local)."


Proceso de integración (Merge)

La integración es el momento en el que el trabajo que has hecho en tu "realidad paralela" (tu rama) se fusiona con la rama principal (main).

Hasta que no haces una integración, tu código solo existe en tu rama. La integración es el paso donde validas que tu trabajo encaja perfectamente con el resto del proyecto sin romper nada.


Merge Local

En este caso no se ha subido la rama a gitHub y eres tú mismo quien verifica que los cambios son correctos


1. Asegurate de estar en la rama destino: git checkout main

2. Trae los últimos cambios de la rama destino: git merge nombre-de-la-rama-con-cambios/code>


Antes de hacer un merge, asegúrate de que tu git status está limpio. Si intentas hacer un merge teniendo archivos modificados sin commit, Git te dará error para proteger tus cambios.


Merge Remoto

Se ha debido subir la rama a gitHub con git pull -u.... Ahora es el profesor o supervisor el que verifica que los cambios son correctos directamente desde GitHub

El siguiente paso es sincronizar el main corregido de gitHub y eleminar las ramas en local

git checkout main

git pull origin main

git branch -d logo-spagnolo-nuevo


Conflictos de integración (Merge Conflicts)

Este es el apartado que más miedo da, pero es parte cotidiana de cualquier desarrollador. Un conflicto ocurre cuando Git no puede decidir por sí mismo qué versión de un archivo es la correcta.


Git te avisará con un mensaje claro: CONFLICT (content): Merge conflict in archivo.js.


    Proceso de resolución

  1. Encuentra el conflicto
    1. <<<<<<< HEAD: Indica el código que tienes en tu rama actual.
    2. =======: Línea divisoria.
    3. >>>>>>> nombre-rama: Indica el código que viene de la rama que intentas fusionar.
  2. Resuelve: Edita el archivo, borra manualmente todas esas marcas (<<<, ===, >>>) y deja el texto como quieres que quede realmente.
  3. Finaliza: bash git add archivo.js git commit -m "fix: resuelto conflicto de fusión entre main y funcionalidad-nueva"
