4. Colaboración profesional

Pull Requests (PRs): La base del trabajo profesional

El Pull Request (PR) es el mecanismo mediante el cual propones formalmente que tus cambios (en tu rama) sean integrados en la rama principal (main). Olvida el enviar archivos por correo o decir "profe, ya funciona"; en DAW, el PR es el único canal oficial para solicitar revisión.


El flujo del PR

  1. Push: Subes tu rama a GitHub mediante git push -u origin <rama>.
  2. Solicitud: En el portal de GitHub, seleccionas tu rama y pulsas el botón verde "Compare & pull request".
  3. Descripción: Escribes brevemente qué has hecho y por qué.
  4. Revisión: El "supervisor" (profesor/compañero) revisa tu código y puede dejar comentarios directamente en las líneas o aprobarlo (Approve).
  5. Merge: Una vez aprobado, el sistema integra (funde) tu código en main

Consejos. "Un Pull Request es una invitación a la revisión. Tu objetivo no es solo pasar código, es facilitar el trabajo a quien debe revisarlo. Un buen PR tiene una descripción clara, cambios atómicos (pequeños) y facilita la comunicación técnica necesaria para mantener el proyecto sano."


Code Review (Revisión de Código)

La Code Review es el proceso de analizar el código de un compañero antes de que se fusione con main. No es un examen ni una crítica; es una herramienta de aprendizaje mutuo y control de calidad.

El objetico principal es dejar comentarios tecnicos en el código, detectando fallos de seguridad (XSS), errores de performance (peticiones innecesarias y falta de legitibilidad)


GitHub Issues: Gestión de Tareas

Un Issue es un ticket de trabajo. Si el Pull Request es la "solución", el Issue es el "problema" o la "necesidad". En DAW, usarás esto para organizar tus prácticas, gestionar errores encontrados y definir nuevas funcionalidades.

  1. ¿Para qué sirven los Issues?
    • Seguimiento de Bugs: "El formulario no envía datos en Firefox".
    • Solicitud de Funcionalidades (Features): "Añadir botón de 'Descargar PDF' en el recibo".
    • Debate: Discutir cómo resolver un problema antes de escribir siquiera una línea de código.
  2. Anatomía de un buen Issue
    • Título descriptivo: Breve y claro.
    • Descripción detallada: Contexto, pasos para reproducir, resultado esperardo, resultado actual.
  3. El Flujo de trabajo profesional (Issue + PR)
    1. Issue: Abres un ticket: "Falta validación en el formulario de login"
    2. Asignación: Te asignas el Issue a ti mismo.
    3. Desarrollo: Creas una rama llamada fix/validar-login.
    4. Vinculación: Cuando abras el Pull Request, escribes en la descripción
