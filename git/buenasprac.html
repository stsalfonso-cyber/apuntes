5. Buenas Prácticas

Commit Messages (Convenciones)

No escribas frases largas. Sigue una estructura predecible. En DAW, te recomiendo usar el estándar "Conventional Commits". Es simple y tu profesor te agradecerá la claridad:

  • Estructura: tipo(ámbito): descripción corta
  • Tipos comunes:
    • feat (feature): Una función nueva
    • fix: Un error corregido.
    • refactor: Cambio de código que no añade ni quita funciones (limpieza).
    • docs: Cambios en los comentarios o README.
  • Ejemplo: feat(auth): añadir validación de email en login

Small Commits (Commit Atómicos)

El error de novato es trabajar 4 horas y hacer un solo commit gigante con 20 archivos modificados. Eso es una bomba de relojería.

  1. ¿Qué es un commit atómico?: Es el commit más pequeño posible que resuelve una sola tarea lógica.
  2. Ventaja: Si algo falla, un commit pequeño es fácil de "deshacer" sin perder todo tu trabajo.
  3. La prueba de ácido: Si tu commit intenta explicar algo y tienes que usar la palabra "y" (Ej: "Añadí el menú y arreglé el pie de página"), es un commit demasiado grande. Divídelo en dos.

GitHub Flow (Tu "hoja de ruta")

El GitHub Flow es el flujo de trabajo más ágil y sencillo del mundo. Si sigues estos 5 pasos, nunca tendrás problemas en DAW:

  1. Todo nace en main: Siempre ten tu main limpio y listo para desplegar.
  2. Crea una rama: Para cada tarea, crea una rama desde main con un nombre descriptivo.
  3. Commits atómicos: Trabaja y haz commits frecuentes en esa rama.
  4. Abre un PR: Cuando acabes, abre el Pull Request. Aquí es donde se discute si el código es apto.
  5. Merge y despliegue: Una vez aprobado, haz merge a main. A partir de aquí, tu cambio ya vive en la rama oficial y está listo para ser publicado.
