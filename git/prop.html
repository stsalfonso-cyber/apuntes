1. El Propósito real: Más allá del backup

Git vs. GitHub: La Distinción Técnica Fundamental

Notas rápidas

  • ◽ Git es el motor local: Un sistema de control de versiones descentralizado que vive en tu disco duro.
  • ◽ GitHub es el centro de colaboración: Una plataforma en la nube (SaaS) construida encima de Git para gestionar equipos y proyectos.
  • ◽ Independencia: Puedes ser un experto en Git y nunca usar GitHub (usando GitLab o Bitbucket), pero no puedes usar GitHub si no sabes Git.
  • ◽ Productividad: Git garantiza tu trabajo offline y rapidez; GitHub garantiza tu visibilidad y trabajo en equipo.

Importante. Son herramientas con propósitos técnicos totalmente diferentes.


1. Git: El Motor Local (El Sistema de Control de Versiones)

Git es una pieza de software que instalas en tu ordenador. Es un VCS (Version Control System) descentralizado.

  • Técnicamente: Es una base de datos local que registra cada cambio en tus archivos.
  • Funcionamiento: Cuando haces un git commit, estás tomando una "foto" (snapshot) de tu proyecto y guardándola en una carpeta oculta llamada .git dentro de tu ordenador.
  • Independencia: Funciona perfectamente sin internet. Puedes hacer commits, ramas y fusiones mientras vas en un tren sin conexión.

2. GitHub: El Centro de Colaboración (La Plataforma en la Nube)

GitHub es una empresa y un servicio web (SaaS) que vive en los servidores de Microsoft.

  • Técnicamente: Es un servidor remoto que "entiende" Git. Aloja tus repositorios y añade una capa superior de herramientas de gestión.
  • Esencial en DAW: GitHub añade lo que Git por sí solo no tiene:
    • Pull Requests (PRs): La forma en que propones cambios para que un profesor o compañero los revise.
    • Gestión de proyectos (Issues/Projects): Organización de tareas.
    • Acciones (CI/CD): Automatización de pruebas y despliegue (la web se publica sola al subir código).
    • Wiki/Documentación: El espacio para que tu proyecto sea legible.

A tener en cuenta:

  1. La trampa de la plataforma: Si crees que "Git es GitHub", pensarás que si GitHub falla, tu trabajo se pierde. Un profesional sabe que, si GitHub cae, su trabajo está seguro en su carpeta .git local.
  2. Seguridad y Privacidad: En un entorno profesional, a veces no puedes usar GitHub por políticas de seguridad (puedes trabajar en un servidor privado usando Git, sin que el código salga de tus servidores). Debes ser capaz de levantar un "Git" propio si el proyecto lo requiere.
  3. Velocidad vs. Colaboración: Entender que Git es para desarrollar localmente (rápido, sin latencia) y GitHub es para colaborar globalmente (lento, sincronización por red) te ayudará a estructurar tus commits.

El inicio, primeros comandos

Fase 1: Inicialización (Solo la primera vez que creas un proyecto)


  • Inicializar localmente: git init (Crea la base de datos interna en la carpeta .git).
  • Configurar identidad: git config --global user.name "Nombre" y git config --global user.email "email" (Registro de quién realiza los cambios).
  • Vincular el servidor remoto: git remote add origin <URL> (Define que el destino oficial de tu trabajo es esa dirección URL)
  • Definir rama principal: git branch -M main (Asegura que tu rama local se llame igual que la de la nube, evitando conflictos de nombre).

Fase 2: El Flujo Diario (Tu rutina de trabajo)


[A] al empezar el día (obtener lo último)

Antes de tocar nada, trae lo último desde GitHub. Si trabajas solo, es buena costumbre. Si trabajas con otros, es obligatorio.


  • git pull origin main: Descarga los cambios de la nube a tu local.

[B] Durante el día (Trabajar)

Escribe código, crea archivos, modifica. Git no hace nada aquí hasta que tú le pides que preste atención


[C] Al terminar el día o al alcanzar un hito (Guardar)

  1. Revisar cambios: git status (Crucial para ver qué ficheros están en la "zona de trabajo" comparados con el disco).
  2. Preparar (Stage): git add . (Mueve los archivos del "área de trabajo" al "área de staging". Están listos para la foto).
  3. Confirmar (Commit): git commit -m "Mensaje descriptivo" (Congela el estado actual. Cada commit es un "punto de guardado" inmutable).
  4. Enviar (Push): git push origin main (Sincroniza tu base de datos local con la nube).

Fase 3: Mantenimiento y diagnóstico

  • git log --oneline: Te da un resumen del historial de cambios (formato corto). Es lo mejor para ver quién subió qué y cuándo.
  • git diff: Si git status te dice que un archivo ha cambiado pero no sabes qué cambió exactamente, esto te muestra línea por línea qué líneas se han editado
  • git branch -a: Por si en el futuro decides crear "ramas" (versiones paralelas de tu código para probar cosas nuevas). Te dirá en qué rama estás trabajando.
  • git remote -v: Te dice a qué URL está vinculado tu proyecto. Muy útil si tienes varios repositorios y te confundes cuál es cuál.

El historial como registro de intent

Si miras el historial de un proyecto de un programador principiante, te encontrarás mensajes de commit como

  • 🔹 "Arreglos"
  • 🔹 "Update"
  • 🔹 "djkfghdkjgfh"
  • 🔹 "Cambios finales (ahora sí definitivo)"

Esta información es totalmente inútil. En seis meses, cuando vuelvas a ese código, no tendrás ni idea de por qué hiciste esos cambios. La filosofía profesional es que tus commits deben ser una narrativa lógica.

Un mensaje de commit profesional responde a tres preguntas: ¿Qué?, ¿Por qué? y ¿Cómo afecta?.

Nomenclaturas estándares par el "¿Qué?"

  • feat: Has añadido una funcionalidad nueva (ej: feat: página de registro).
  • fix: Has arreglado un error (ej: fix: corregido cálculo IVA en carrito)
  • refactor: Has mejorado el código sin cambiar lo que hace (limpieza).
  • docs: Has cambiado documentación.

Ejemplo práctico

fix(carrito): se ajusta validación para campos vacíos antes del envío.

Se detectó que el usuario podía enviar el formulario de compra con campos vacíos, causando errores en el servidor. Se añade validación previa para prevenir el envío si los datos son inválidos.


"La regla de oro del Commit": "Escribe cada mensaje de commit como si el destinatario fuera tu 'yo' del futuro dentro de 6 meses". Si él no entiende qué hiciste y por qué, el commit no es bueno.


Trabajar en equipo

Para que 10 personas trabajen en el mismo código sin pisarse, Git utiliza tres pilares: Ramas (Branches), Resolución de Conflictos y PRs (Pull Requests).


  1. 1. Ramas (Branches): Tu "zona de pruebas"

    En Git, nadie trabaja directamente sobre el código "oficial" (la rama main o master).

    • ◾Creas un "paralelo": Cuando empiezas una tarea, haces una ramificación. Es como sacar un "clon" del proyecto.
    • ◾Aislamiento: Puedes romper lo que quieras en tu rama que el resto del equipo ni se entera. El código principal sigue intacto y funcionando.
    • ◾Merge: Una vez que tu tarea está perfecta y probada, "fusionas" (haces merge) tu rama con la principal.

  2. 2. Conflictos: Cuando Git dice "¡Alto!"

    Cuando dos personas tocan la misma línea del mismo archivo y Git no sabe cuál de las dos versiones es la correcta, Git marcará los conflictos con unas etiquetas, el usuario eligirá que opcion permanecerá y Git lo registrará


  3. 3. El Pull Request (PR): El filtro de calidad

    Cuando la rama está terminada, antes de realiza el proceso merge, abres un Pull Request, todos los compañeros que comparten el trabajo tienen que aprobar el cambio primero


    4. Resumen: "La colaboración profesional en DAW se basa en no trabajar NUNCA en la rama principal (main). Para cada tarea, creas una rama, trabajas aislado, abres un Pull Request para revisión, y solo cuando el código es aprobado, se fusiona (merge) en el proyecto principal."


Marca Personal: Tu vitrina profesional

GitHub no es un "servidor de archivos", es tu portfolio técnico. En el entorno DAW, un perfil bien cuidado es el primer filtro que supera un candidato antes incluso de la entrevista personal.


El Perfil (Landing Page)

Cualquier usuario en GitHub tiene un repositorio especial con su nombre de usuario.

El README.md de este repositorio es tu página de aterrizaje.

  • ◻ Debe contener: Quién eres, qué tecnologías dominas (JavaScript, DOM, Git…), en qué proyecto emocionante estás trabajando ahora e información de contacto profesional.
  • ◻ La regla del "cuadro verde": La actividad (los cuadraditos verdes de tu perfil) demuestra constancia. No busques perfección, busca frecuencia. Un estudiante activo es un estudiante que aprende.

Plantilla profesional para todo repositorio

Nunca subas un repositorio vacío o sin explicaciones. Todo proyecto que subas a GitHub como estudiante de DAW debe llevar un README.md con esta estructura mínima:



    # Nombre del Proyecto
Breve descripción: ¿Qué problema resuelve este código?

## Tecnologías utilizadas
- JavaScript (ES6+)
- DOM API
- Fetch API

## Cómo ejecutarlo
1. Clona este repositorio: `git clone [url]`
2. Abre `index.html` en tu navegador.

## Aprendizajes clave
- Gestión de promesas con async/await.
- Manipulación del DOM mediante querySelector.

Checklist de Higiene Profesional

Para que tu perfil luzca como el de un desarrollador senior desde el primer día:

  • ◻ Naming: Nombres descriptivos y en minúsculas (ej: spagnolo-buscador-productos, no proyecto1 o test).
  • ◻ .gitignore real: Asegúrate de que no estás subiendo archivos temporales, contraseñas o carpetas gigantes (node_modules).
  • ◻ Licencia: Añade siempre un archivo LICENSE. Aunque sea pequeña, una licencia tipo MIT indica que te preocupas por el aspecto legal y profesional de tu trabajo.
  • ◻ Small Commits: Un repositorio profesional tiene decenas de commits pequeños y atómicos, no dos o tres commits gigantes que dicen "subiendo proyecto".
