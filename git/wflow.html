2. El "Workflow" de supervivencia (Git básico)

Configuración Inicial

Identidad: git config

GitHub necesita asociar tus commits con tu cuenta. Si no haces esto, tus aportaciones aparecerán como "anónimas".


# Configura tu nombre (se verá en el historial)
git config --global user.name "Tu Nombre Real"

# Configura tu email (DEBE ser el mismo que usas en GitHub)
git config --global user.email "tu-email-de-github@ejemplo.com"

--global: Significa que esta configuración se aplicará a todos los repositorios de tu ordenador. Si algún día trabajas en un proyecto de empresa con un email distinto, podrás usar --local solo para ese proyecto.


Seguridad: Autenticación con SSH

Olvídate de introducir tu contraseña cada vez que haces un push. Las claves SSH permiten que tu ordenador y GitHub se "reconozcan" de forma segura mediante un par de llaves: una pública (que se la das a GitHub) y una privada (que vive solo en tu ordenador y nunca se comparte).


1 - Generar la clave:

ssh-keygen -t ed25519 -C "tu-email-de-github@ejemplo.com"

(Pulsa "Enter" a todo para aceptar las rutas por defecto. No hace falta poner contraseña a la clave si estás en un ordenador personal seguro).


2 - Copiar tu clave pública:

En Windows (Git Bash): clip < ~/.ssh/id_ed25519.pub (esto copia el contenido al portapapeles).

En Mac/Linux: cat ~/.ssh/id_ed25519.pub (copia el texto que sale en pantalla).


3 - Vincularla a GitHub:

  • Ve a tu perfil de GitHub > Settings (esquina superior derecha).
  • En el menú lateral izquierdo, busca SSH and GPG keys.
  • Haz clic en New SSH Key.
  • Ponle un título (ej: "Portátil DAW") y pega el texto que copiaste en el paso anterior.

4 - Verifica la conexión:

ssh -T git@github.com

Si te sale un mensaje diciendo: "Hi [TuUsuario]! You've successfully authenticated…", ¡felicidades!, ya tienes seguridad de grado profesional.


Tu rutina diaria (El ciclo dorado)

  1. git status - El mapa de situación

    Es el primer comando que debes ejecutar siempre. Te dice: ¿Qué archivos has modificado? (Aparecen en rojo). ¿Están siendo rastreados por Git? Es tu brújula: Nunca tomes una decisión sin saber qué archivos tienes tocados.

  2. git add <archivo> — La Sala de Espera (Staging Area)

    Añade cada archivo que quieras que se suba a GitHub. Con git add . especificas todos los archivos modificados y nuevos del directorio actual.

  3. git commit -m "mensaje" — La Foto

    Aquí es donde registras el cambio de forma permanente en la historia.

    Una vez haces el commit, ya no hay marcha atrás para ese registro.

    La importancia del mensaje: Como vimos en el punto anterior, aquí es donde explicas el "por qué" (el intent).

    Nota: Este comando solo guarda el cambio en tu ordenador local. Si se rompe tu portátil, esto no está en GitHub todavía.

  4. git push origin main — El Envío (Sincronización)

    Sube tus commits locales al servidor remoto en GitHub. Ahora tu código está seguro en la nube y tus compañeros o profesores pueden verlo.

    Nota: "origin" hace referencia al servidor remoto y "main" a la rama principal de trabajo.


Recuperación y sincronización

1. git clone — El punto de partida

Se usa una sola vez por proyecto. Es el comando que ejecutas cuando vas a descargar el código base de un repositorio de GitHub para empezar a trabajar en tu ordenador por primera vez.

git clone https://github.com/usuario/nombre-repo.git

Descarga el historial completo, todas las ramas y todos los archivos del repositorio remoto y crea una carpeta local configurada como un repositorio de Git listo para trabajar.


2. git pull — El "Ciclo de Vida Diario"

Este comando se trae los cambios que están en el servidor remoto pero no están aplicados en tu proyecto local, actualizando tu rama de trabajo de forma automática.

Este proceso también se puede hacer de manera manual en dos partes:

  • git fetch: Descarga del servidor remoto la información de los nuevos cambios sin modificar tus archivos locales.
  • git merge: Aplica y fusiona esos cambios descargados en tu rama de trabajo actual.

El olvidado .gitignore

El archivo .gitignore es un archivo de texto simple que le dice a Git qué archivos o carpetas debe ignorar y nunca rastrear ni subir a GitHub. Esto es fundamental para mantener tu repositorio limpio, profesional y seguro.


¿Qué debemos ignorar siempre?

  • Carpetas pesadas de dependencias (ej: node_modules/ en Node.js, vendor/ en PHP).
  • Archivos de configuración personal o del IDE (ej: .vscode/, .idea/, archivos de estado del sistema).
  • Archivos del sistema operativo (ej: Thumbs.db en Windows, .DS_Store en macOS).
  • Credenciales, tokens y contraseñas (ej: archivos .env). Subirlos a GitHub es un peligro gravísimo para la seguridad de tu proyecto.

Ejemplo práctico de un archivo .gitignore


# Ignorar dependencias de Node.js
node_modules/

# Ignorar archivos del sistema operativo
.DS_Store
Thumbs.db

# Ignorar variables de entorno (contraseñas, claves API)
.env
config.private.json

# Ignorar carpetas de configuración de editores de código
.vscode/
.idea/
