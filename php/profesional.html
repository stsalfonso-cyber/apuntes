Composer: Gestión de Dependencias (El "npm" de PHP)

Composer es el gestor de dependencias estándar en PHP. Si vienes de JavaScript, piensa en él como npm o yarn. Es una herramienta que permite instalar librerías externas (código escrito por otros) de forma fácil, segura y automática en tus proyectos.

Paso a paso:

1. Iniciamos Composer: composer init

Esto crea un archivo composer.json en tu carpeta. Es como el "carnet de identidad" de tu proyecto, donde se apuntará todo lo que instales.

2. Instalamos una librería (ej: Faker): composer require fzaninotto/faker

    ¿Qué ha pasado?
  1. Composer ha conectado con el "App Store" de PHP.
  2. Ha descargado faker en la carpeta /vendor.
  3. Ha actualizado el composer.json para recordar que ahora este proyecto depende de faker
  4. Ha creado el archivo vendor/autoload.php (sta carpeta va en el gitignore).

Usamos la librería en tu código:

    
<?php
// 1. Cargas el autocargador (solo se hace una vez al inicio)
require 'vendor/autoload.php';

// 2. Usas la librería directamente
$faker = Faker\Factory::create();

// Genera datos falsos al instante
echo $faker->name;      // "Juan Pérez"
echo $faker->email;     // "juan@example.com"
echo $faker->address;   // "Calle Falsa 123"

Autoloading: La carga inteligente de archivos

El Autoloading evita tener que cargar decenas de archivos con require o include. lo automatiza. detecta cuándo intentas usar una clase que aún no se ha cargado y la busca y carga en el servidor automáticamente por ti.

  1. El concepto: "Lazy Loading" (Carga bajo demanda)

    El Autoloading funciona bajo la filosofía: "No cargues nada en memoria hasta que sea estrictamente necesario". Esto ahorra memoria RAM del servidor y evita conflictos de carga.

  2. El estándar profesional: PSR-4

    El mundo PHP se puso de acuerdo en un estándar llamado PSR-4. La regla es simple: Tu carpeta y tu namespace deben hablar el mismo idioma.

      Si tu clase se llama Spagnolo\Controllers\Usuario y sigue la regla PSR-4:
    • PHP buscará automáticamente en la carpeta /Spagnolo/Controllers/Usuario.php.
    • Si el archivo no existe en esa ruta exacta, PHP te dará error.
  3. La magia de Composer

    ¿Cómo sabe PHP dónde está cada archivo si yo no uso require?

    Aquí entra la magia de Composer. Cuando ejecutas composer require..., Composer analiza todos los archivos que ha descargado y crea un mapa (o índice) en el archivo vendor/autoload.php.

    Cuando tú escribes require 'vendor/autoload.php'; en tu script principal, lo que realmente haces es inyectar un "buscador profesional" que consulta ese mapa y sabe exactamente dónde está cada clase que necesites.

    Resumen del proceso:
  1. Instalas una librería con Composer (ej: fakerphp/faker).
  2. No haces include de nada de faker.
  3. Haces require 'vendor/autoload.php'; al inicio.
  4. Usas la clase: new Faker\Factory();

Estándares PSR (PHP Standards Recommendations)

Los PSR son una serie de recomendaciones publicadas por el PHP-FIG (PHP Framework Interoperability Group). Su objetivo es simple: que tú puedas escribir código, y que cualquier otro desarrollador en el mundo pueda entenderlo al instante.

Estos son los tres estándares existentes mas importantes y pilares en DAW

  1. El pilar de la estructura: PSR-4 (Autoloading)

    Ya lo hemos visto: Namespace = Rutas de carpetas.

    ◽ Si tu clase es Spagnolo\Modelos\Usuario, Composer debe buscarla en src/Modelos/Usuario.php.

    ◽ Aceptar este estándar es lo que permite que el vendor/autoload.php funcione. Si rompes el PSR-4, el autoloading se rompe.


  2. El pilar del estilo: PSR-1 y PSR-12 (Coding Style)

    Estas son las normas de "limpieza":

    ◽ PSR-1 (Basic): Define que PHP debe usar <?php y ?> (o solo <?php), que los nombres de clase deben ser StudlyCaps (Ej: MiClasePro), y los nombres de constante UPPER_CASE.

    ◽ PSR-12 (Extended): Es el estándar de formato más usado actualmente. Define cosas como:

    1. - Indentación: 4 espacios (no tabulaciones).
    2. - Llaves: Las llaves de apertura de una clase van en una línea nueva.
    3. - Espacios: Obliga a poner un espacio después de una palabra clave como if o foreach.

  3. El estándar de facto: ¿Por qué seguirlo?

    1. Interoperabilidad: Si usas librerías de terceros (como faker o dompdf), todas ellas siguen PSR. Si tu código también lo sigue, todo "encaja" perfectamente.
    2. Lectura: La mayoría de los proyectos empresariales en PHP (Laravel, Symfony) aplican PSR-12 automáticamente. Si escribes a tu manera, serás un "cuerpo extraño" en el equipo.

Uso de herramientas automáticas

  • PHP-CS-Fixer: Es una herramienta que instalas en tu proyecto (con Composer) que leí tu código y lo corrige automáticamente para cumplir con PSR-12.

    ▫ Tú solo escribes código; ejecutas la herramienta y ella corrige cada espacio, cada llave y cada salto de línea

  • Plugins de VS Code: Instala una extensión llamada php-cs-fixer o PHP Intelephense. Configurándolas correctamente, tu código se formatea solo al darle a guardar (Ctrl+S).
