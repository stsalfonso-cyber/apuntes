[5] Acceso a Datos y Seguridad

PDO (PHP Data Objects)

Las funciones mysql_query y sqlsrv_query han querdao totalmente obsoletas. El estándar absoluto es PDO (PHP Data Objects).

Si usas PDO puedes cambiar de SQL Server a MySQL simplemente cambiando una línea de configuración

1. El secreto: El DSN (Data Source Name)

El DSN es una simple cadena de texto que le dice a PHP: "¿A qué base de datos me conecto y dónde está?".

Para MySQL

$dsn = "mysql:host=localhost;dbname=nombre_db;charset=utf8mb4";

Para SQL Server

$dsn = "sqlsrv:Server=localhost;Database=nombre_db";

(Nota: Aunque el driver cambie, el resto de tu código PHP será exactamente el mismo.)


2. El proceso SELECT (traer datos)

    
try {
    // 1. Conexión  ( ESTO SUELE ESTAR EN UN ARCHIVO APARTE config.php o db.php )
    $pdo = new PDO($dsn, "usuario", "password");
    $pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);

    // 2. Preparar (La plantilla)
    // Usamos :nombre en lugar del valor real (esto evita SQL Injections)
    $stmt = $pdo->prepare("SELECT * FROM productos WHERE categoria = :categoria");

    // 3. Ejecutar (Pasando los valores para el filtro)
    $stmt->execute(['categoria' => 'camisas']);

    // 4. Recuperar (fetchAll recupera toda la tabla en un array asociativo)
    $productos = $stmt->fetchAll(PDO::FETCH_ASSOC);

} catch (PDOException $e) {
    echo "Error de conexión: " . $e->getMessage();
}

// Esto cierra la conexion completamente, aunque no es necesario 
// porque se cierra automáticamente cuando el script termina
$pdo = null;

3. La esencia: ¿Por qué protege contra SQL Injection?

  • La forma "amateur" (Peligrosa): 
    $sql = "SELECT * FROM usuarios WHERE email = '" . $_POST['email'] . "'";

    Un hacker pone un email falso tipo: inventado@falso.com' OR '1'='1

    El motor SQL interpretará esto SELECT * FROM usuarios WHERE email = 'inventado@falso.com' OR '1'='1' y entra sin contraseña ya que la condicion 1 = 1 ES CIERTA.

  • La forma PDO (Segura):

    Tú envías una "Plantilla" (la query) y los datos por separado. La base de datos recibe la estructura primero, y luego los datos se "pegan" de forma segura donde van los :marcador. El motor de SQL sabe perfectamente que :categoria no es código ejecutable, sino un simple valor de texto. Es imposible que un hacker inyecte código porque el servidor ya no interpreta "datos" como "comandos".


4. Elegir while + fetch sobre foreach + fetchAll

La desición de elegir entre uno u otro recae en la cantidad de datos que contenga la tabla.


Cuando hacemos $productos = $stmt->fetchAll(PDO::FETCH_ASSOC); recuperamos toda la tabla en una variable que se puede recorrer con foreach

    
$productos = $stmt->fetchAll(PDO::FETCH_ASSOC);
// $productos ahora es así:
// [
//     0 => ['id' => 1, 'nombre' => 'Polo Azul', 'precio' => 45.00],
//     1 => ['id' => 2, 'nombre' => 'Polo Rojo', 'precio' => 45.00],
//     2 => ['id' => 3, 'nombre' => 'Camisa Oxford', 'precio' => 60.00]
// ]

foreach ($productos as $producto) {
    
    // En la primera vuelta del bucle:
    // $producto se convierte en esto: ['id' => 1, 'nombre' => 'Polo Azul', 'precio' => 45.00]

    // Por eso aquí NO necesitas el [0]. 
    // $producto YA ES el array de la fila correspondiente.
    echo $producto['nombre']; 
}

El peso de la variable producto es el de toda la base de datos, si es muy grande puede colapsar la memoria RAM


Pero ahora hacemos $producto = $stmt->fetch(PDO::FETCH_ASSOC) dentro de un while, esto resupera uno por uno cada registro de la tabla y lo convierte en un array asociativo sólo de este registro

    
while ($producto = $stmt->fetch(PDO::FETCH_ASSOC)) {
    echo $producto['nombre'];
// ahora este es valor del array asociativo producto en la primera vuelta:
// ['id' => 1, 'nombre' => 'Polo Azul', 'precio' => 45.00]

}

El peso de la varible producto no será tan elevado ya que solo recupera un registro en cada iteración del while


5. El proceso INSERT (añadir datos)

    
// 1. Conexión
$pdo = new PDO($dsn, "usuario", "password");
$pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);

// 2. Preparar
$sql = "INSERT INTO productos (nombre, precio, categoria) VALUES (:nombre, :precio, :categoria)";
$stmt = $pdo->prepare($sql);

// 3. Ejecutar
$stmt->execute([
    'nombre'    => 'Polo Spagnolo',
    'precio'    => 45.00,
    'categoria' => 'hombre'
]);

// 4.Recuperar el ID que se acaba de generar automáticamente
$nuevoId = $pdo->lastInsertId();
echo "Producto creado con ID: " . $nuevoId;

6. El proceso UPDATE (actualizar datos)

    
// 1. Conexión
$pdo = new PDO($dsn, "usuario", "password");
$pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);

// 2. Preparar
$sql = "UPDATE productos SET precio = :nuevo_precio WHERE id = :id";
$stmt = $pdo->prepare($sql);

// 3. Ejecutar
$stmt->execute([
    'nuevo_precio' => 39.99,
    'id'           => 12
]);

// 4.Recuperar el total de lineas actualizadas para validar
if ($stmt->rowCount() > 0) {
    echo "Producto actualizado correctamente.";
} else {
    echo "No se encontró el producto o no había cambios.";
}

6. El proceso DELETE (borrar datos)

    
// 1. Conexión
$pdo = new PDO($dsn, "usuario", "password");
$pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);

// 2. Preparar
$stmt = $pdo->prepare("DELETE FROM productos WHERE id = :id");

// 3. Ejecutar
$stmt->execute(['id' => 12]);

// 4.Valida la eliminacion
if ($stmt->rowCount() > 0) {
    echo "Producto eliminado.";
} else {
    echo "El producto ya no existía.";
}

Seguridad Web: La "Santísima Trinidad"

La seguridad en PHP se resume en "nunca confíes en la entrada de datos". Si el dato viene del usuario, es potencialmente peligroso.


Inyección SQL (SQL Injection)

Salió en el tema PDO pero reafirmamos la regla de oro

  • El problema: Concatenar variables directamente en la cadena SQL. Un atacante introduce comandos SQL en los campos de texto.
  • La defensa: Sentencias preparadas (PREPARE + :marcadores).
  • Manual: Nunca uses query() con variables concatenadas. Usa siempre prepare() y pasa los valores a través del array de execute(). Es el único blindaje real certificado ante ataques de inyección.

XSS (Cross-Site Scripting)

  • El problema: Un atacante introduce código JavaScript en un formulario (ej: <script>alert('hack')</script>). Cuando tú haces un echo de ese dato en tu HTML, el navegador del usuario ejecuta ese script malicioso.
  • La defensa: Sanitización de salida. Nunca muestres datos crudos.
  • Manual: Usa siempre la función htmlspecialchars($data) antes de imprimir cualquier dato que venga del usuario.
    
// Mal: si el usuario escribe <script>, este se ejecuta
echo $comentario_usuario; 

// Bien: lo convierte en &'lt;script&'gt;, texto inofensivo
echo htmlspecialchars($comentario_usuario, ENT_QUOTES, 'UTF-8');

CSRF (Cross-Site Request Forgery)

  • El problema: Un atacante engaña a un usuario logueado en tu web para que haga clic en un enlace malicioso que envía una petición (ej: borrar su cuenta o cambiar la contraseña) sin que el usuario se dé cuenta.
  • La defensa: Tokens CSRF.
  • Manual: Es un "pase de seguridad" único y secreto generado por el servidor.
    1. El servidor genera un token aleatorio al cargar el formulario y lo guarda en $_SESSION.
    2. El formulario incluye ese token en un campo oculto (<input type="hidden" name="csrf_token" value="...">).
    3. Al recibir el post, el servidor compara el token del formulario con el de $_SESSION. Si no coinciden, la petición es rechazada.
    
<?php
session_start();
// Generamos un token aleatorio y seguro
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32)); 
}
?>

<!-- En tu formulario HTML, añades un campo oculto -->
<form action="procesar.php" method="POST">
    $lt;input type="hidden" name="csrf_token" value="<?php echo $_SESSION['csrf_token']; ?>">
    $lt;input type="text" name="email">
    $lt;button type="submit">Enviar
</form>

Y la validación...

    
<?php
session_start();

// Validamos el token
if (!isset($_POST['csrf_token']) || !hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'])) {
    // Si no coincide, TERMINAMOS TODO. Es un intento de hackeo.
    die("Error de seguridad: Solicitud CSRF inválida.");
}

// Si llega aquí, el token es correcto, ya podemos procesar el resto
echo "Datos procesados con éxito.";
?>

Hashing de Contraseñas (Nunca guardes claves planas)

Es el pecado capital del desarrollo backend: guardar contraseñas como texto ("123456").

  • El problema: Si te hackean la base de datos, el hacker tiene todas las contraseñas de tus usuarios.
  • La defensa: Utiliza funciones de hash modernas (no uses md5 ni sha1, son antiguas).
    
// Al guardar:
$hash = password_hash($password, PASSWORD_DEFAULT);

// Al validar:
if (password_verify($password, $hashEnBaseDeDatos)) {
    echo "Contraseña correcta";
}

Tu Checklist de Seguridad

  1. SQL: ¿Usas prepare()?
  2. XHTML/Frontend: ¿Has usado htmlspecialchars() en todo lo que imprimes de un usuario?
  3. Formularios: ¿Has implementado tokens CSRF?
  4. Passwords: ¿Guardas los hashes con password_hash()?
