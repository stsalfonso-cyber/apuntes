Superglobals: El Puente Frontend-Backend

Las Superglobals son el corazón de la comunicación en PHP. Se llaman "superglobales" porque son variables predefinidas que están siempre disponibles en cualquier parte del código (dentro de funciones, clases, archivos importados, no importa dónde).


Representan los dos mundos de la web: lo que viene del cliente (entradas) y lo que el servidor conoce (entorno).


1. Las Superglobals de entrada (Input)

Estas son las que leerás constantemente cuando el usuario interactúa. Su nombre siempre empieza por $_ y deben escribirse siempre en mayúsculas.

  • $_GET: Captura los parámetros que viajan en la URL.
    • Ejemplo: spagnolo.es/perfil.php?id=10
    • En PHP: $id_producto = $_GET['id'];
  • $_POST: Captura los datos enviados por un formulario cuando el método es method="POST".
    • Es más seguro y permite enviar más datos que $_GET
    • En PHP: $nombre = $_POST['nombre_usuario'];
  • $_REQUEST: Es una mezcla de GET, POST (y COOKIE). Al principio parece cómoda, pero no se recomienda en desarrollo profesional porque es menos específica y puede causar confusión.

    • 2. La lógica de "La Red de Seguridad" (El operador ??)

    Como aprendimos, nunca puedes confiar en que el usuario haya enviado el dato. Si intentas leer $_GET['id'] y el usuario no lo ha puesto, tu app disparará un error Undefined index.

    La forma correcta de leer supperglobals es:

        
// Si 'id' no existe, te devuelve null y no peta
$id = $_GET['id'] ?? null; 

// Si 'nombre' del formulario no existe, le pones un valor por defecto
$nombre = $_POST['nombre'] ?? 'Invitado';

    3. Las Superglobals de entorno y sesión

    • $_SESSION: ¡Fundamental! Como PHP "muere" al terminar la petición (lo que hablamos en el apartado 1), esta variable es la única manera de recordar cosas entre páginas (ej: que el usuario ha hecho login)

      En PHP cuando termina de generar una página, el script se destruye y olvida todo. $_SESSION es la solución: es una variable global que vive en el servidor (EN LA MEMORIA RAM DEL SERVIDOR) y que sobrevive de una página a otra mientras el usuario está navegando.

      Se pueden gardar cualquier tipo de dato, String, integer, booleano, array y objeto

              
<?php
session_start(); // ¡Sin esto, PHP no sabe de quién es la sesión!

// 1. Guardar cualquier dato
$_SESSION['usuario_id'] = 101;
$_SESSION['carrito'] = ['id' => 5, 'precio' => 20];
$_SESSION['config'] = new UsuarioConfig(); // Puedes guardar objetos

// 2. Leer datos en otra página (ej: perfil.php)
if (isset($_SESSION['usuario_id'])) {
    echo "Hola usuario " . $_SESSION['usuario_id'];
} else {
    echo "No has iniciado sesión";
}

// 3. Destruir sesión (Logout)
session_unset(); // Borra todas las variables almacenadas
session_destroy(); // Mata la sesión del servidor
?>
    • $_SERVER: Contiene información técnica del servidor y del navegador del usuario
        Por Ejemplo.
      • $_SERVER['REQUEST_METHOD'] te dice si el usuario entró vía GET o POST.
      • $_SERVER['REQUEST_URI']: La ruta completa de la página (ej: /productos/camisa-azul?color=azul).
      • $_SERVER['QUERY_STRING']: Todo lo que va después del signo ? en la URL.
      • $_SERVER['REMOTE_ADDR'] te da la dirección IP del usuario.

    Gestión de Persistencia (Sesiones vs Cookies)

    Aunque en el día a día las usamos juntas, son conceptos técnicos totalmente distintos. La diferencia fundamental está en dónde se guarda la información.


    Cookies: El "Sticker" en el navegador del usuario

    ▪ Sintaxis: 

        
// los parametros se pasan en este orden
//   (nombre, valor, expiración, path) 
setcookie("preferencia_idioma", "es", time() + (86400 * 30), "/"); // + (86400 * 30) significa 30 días de duración
// para acceder al valor
$idioma = $_COOKIE['preferencia_idioma'] ?? 'es';

    ▪ Lugar: Disco duro del usuario (o memoria del navegador).

    ▪ Seguridad: Baja. Cualquier usuario puede ver sus cookies desde las herramientas de desarrollador. Nunca guardes contraseñas o datos críticos aquí.

    ▪ Recuperación: Puedes leerlas usando la superglobal $_COOKIE.

    echo $_COOKIE['preferencia_idioma'] ?? 'es';

    Sesiones: La "Caja fuerte" en tu servidor

    ▪ Sintaxis: 

        
session_start(); 
$_SESSION['usuario_id'] = 45;

    ▪ Lugar: Servidor (Memoria RAM o disco del servidor).

    ▪ Seguridad: Alta. El usuario nunca ve los datos de la sesión, solo ve el identificador (PHPSESSID).


    ¿Cuál elegir? Si el dato es sensible (quién es el usuario, qué ha comprado), usa $_SESSION. Si el dato es cosmético o de configuración (qué color de fondo prefiere, idioma), usa setcookie().

    ◾ Ademas, el uso de SESSION tambien utiliza cookies, la id de la sesion se queda guardada en una cookie para que el navegador sepa quien ha iniciado sesion y permitirle el acceso

    Headers

    Lo que realmente usaras

    1. Redirecciones: 
              
header("Location: dashboard.php");
exit(); // ¡OBLIGATORIO siempre tras un Location!
    2. APIs y JSON (Backend moderno): 
              
header("Content-Type: application/json; charset=UTF-8");
echo json_encode($datos);
    3. Control de Caché (Rendimiento): 
              
header("Cache-Control: no-cache, no-store, must-revalidate");

    "Los HTTP Headers son metadatos que definen la 'naturaleza' de la respuesta."

    • Orden: Siempre van al principio (antes de cualquier echo).
    • Redirección: header('Location: ...') + exit() es la combinación ganadora para flujos de login/logout.
    • API: Cada vez que envíes JSON, debes avisar al navegador con Content-Type: application/json.
    • Prevención: Si recibes el error "Headers already sent", busca el culpable: un espacio en blanco antes de <?php o un echo mal puesto al principio del archivo.

    ¿Por qué son necesarias?

    Porque te permite tomar el control total. Sin cabeceras, el servidor solo "suelta texto". Con cabeceras, tú decides:

    • Si el usuario puede ver la página o tiene que irse a otra parte (Location).
    • Si el navegador debe descargar un archivo, mostrarlo o guardarlo (Content-Disposition).
    • Si el navegador debe confiar en tus datos o no.
